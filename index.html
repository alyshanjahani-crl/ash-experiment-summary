ASH Query
SELECT work_event_type, work_event,
       count(*) AS samples
FROM crdb_internal.cluster_active_session_history
WHERE app_name = 'kv'
  AND workload_type = 'STATEMENT'
  AND sample_time BETWEEN '16:44' AND '16:50'
GROUP BY 1, 2 ORDER BY samples DESC;

Key Insight: SQL stats and the contention graph tell you there is contention, but not what kind. ASH reveals it is LockWait (14,164 samples — transaction-level lock conflicts on hot keys), not LatchWait (1,401 — internal serialization). This distinction matters: LockWait means application-level key distribution is the problem.

Key Takeaways

ASH Complements Existing Observability

Root Cause Identification

DB Console shows contention exists; ASH reveals it is LockWait (hot keys) vs. LatchWait (serialization) — directly actionable

Workload ID Attribution

ASH attributes admission events to specific workload_ids — the exact statement fingerprint or job ID being throttled, not just aggregate metrics

Job-Level Visibility

Jobs page shows status/duration; ASH reveals where each job spends its time (CPU, IO, admission, network) by exact job ID

System Work Proportion

ASH shows system work as a fraction of total cluster activity — 27.6% during heavy writes, enabling full-picture analysis

SQL-Queryable

All data available via crdb_internal.cluster_active_session_history — slice and dice with standard SQL, no external tooling needed

Summary

ASH answers "where is time being spent?" at a granularity that metrics dashboards cannot. Its key differentiator is workload_id attribution — attributing work to specific statements, jobs, and system tasks with proportional breakdowns across event types.

Together with existing DB Console metrics, ASH provides a complete picture: metrics show what happened, ASH shows why.
